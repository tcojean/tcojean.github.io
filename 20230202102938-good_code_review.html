2. Code reviews

  • Essential to code quality
  • Ensure knowledge sharing
  • Allows to confirm or review the global code structure, design
  • Review portions of the code depending on area of expertise

2.1. Support

  • Automated code coverage (report in the PR)
  • Automated quality checks (on the gitlab)

3. Conducting reviews

  • Our PRs are usually big (>1k lines)
  • It’s fine to only do portions at a time
  • Or review only a small part
    • tests
    • interfaces
    • algorithms
    • OneAPI
  • We have interest groups, see https://github.com/orgs/ginkgo-project/teams/reviewers/teams

3.1. Process

Many approaches, depend on review objective. What I and others usually do:

  1. A light pass to find the purpose of the PR and its scope: description, interfaces, examples/tests
  2. If something prevents clear reading of the PR, mention to the author and wait.
  3. Another light pass for the context: how does this relate to the rest of the code, other efforts (past, ongoing or future?). Design issue popping up?
  4. Diving into the code

3.1.1. Diving into the code

  • Start with tests or examples to understand functionality usage
  • Look at interfaces, general logic, requirements and dependencies. “main” aspects of the PR.
  • Algorithms: data initialization.
  • Algorithms: core functionality.
  • Come back to tests, do they match the algorithms? Check corner cases?

3.2. Things to look out for

  • Usability
  • Missing or wrong documentation is a bug
  • Needless complexity is a bug in waiting
  • Missing tests
  • Corner cases, empty data, non-tested branches, …
  • Errors, exceptions are appropriate and are tested

3.3. Writing the review

  • Stay civil: the code is being reviewed not the person
  • Distinguish important comments from nitpicks
  • Summarize in the review text your main concerns, and explain why
  • Stay open ended, offer alternatives or workarounds if possible
  • Don’t hesitate to ask questions in the review: if you don’t understand, documentation is missing.

3.4. Writing the review (bis)

  • No need to aim for perfection, code development is an iterative process.
  • If a question or concern look bigger, after discussing with the author, create an issue out of it, bring it in a meeting, open polls, …
